הכנה למבחן: פיתוח יישומי אינטרנט

גרסה מורחבת ומעמיקה הכוללת את התיאוריה המלאה מאחורי כל נושא וניתוח מדויק של שאלות הבחינה. מומלץ לקרוא את תקצירי העומק לפני המענה על השאלות.

1. שפות צד לקוח: תשתית, מבנה ועיצוב (HTML5 & CSS3)

הבסיס: HTML ומרכיביו

HTML (HyperText Markup Language) אינה שפת תכנות (אין בה לולאות, משתנים או תנאים לוגיים), אלא שפת תיאור ומבנה. היא משתמשת בתגיות (Tags) כדי להגדיר לדפדפן מהו כל רכיב בעמוד (פסקה, כותרת, טופס, תמונה). לכל תגית יכולים להיות מאפיינים (Attributes) כגון `class`, `id`, `src` או `href` אשר מעניקים מידע נוסף על האלמנט או מגדירים את התנהגותו.

אלמנט DIV מול אלמנטים סמנטיים (HTML5)

התגית <div> היא אלמנט מסוג Block-level. המשמעות היא שהיא תמיד תתחיל בשורה חדשה ותתפוס את כל הרוחב הזמין (100% מרוחב אלמנט האב שלה). כשלעצמה, לתגית זו אין שום משמעות סמנטית - היא משמשת רק כ"מיכל" (Container) ארגוני שנועד לקבץ אלמנטים יחד כדי להחיל עליהם עיצוב CSS או מניפולציות ב-JS.

גרסת HTML5 עשתה מהפכה בכך שהציגה תגיות סמנטיות (כגון <header>, <footer>, <article>, <nav>). תגיות אלו מסבירות למנועי החיפוש (SEO) ולקוראי מסך (נגישות) את המשמעות של התוכן, ולא רק את המבנה שלו. בנוסף, HTML5 הביאה עימה ממשקי API מובנים (כמו Local Storage, Video/Audio מקומיים ו-Canvas לציור) שביטלו את הצורך בתוספים חיצוניים כמו פלאש.

עיצוב רספונסיבי וספציפיות ב-CSS

עיצוב רספונסיבי (Responsive Design): גישה בה ממשק המשתמש מתאים את עצמו באופן אוטומטי לכל גודל מסך או מכשיר (דסקטופ, טאבלט, מובייל). הכלים המרכזיים למימוש עיצוב כזה הם יחידות מידה יחסיות (%, vw, vh, rem) וחשוב מכל - Media Queries. פקודות אלו (למשל `@media (max-width: 768px)`) מאפשרות לדפדפן להחיל חוקי CSS שונים כאשר תנאי המסך משתנים.

ספציפיות (Specificity) ב-CSS: מערכת הניקוד של הדפדפן שקובעת איזה עיצוב יוחל במקרה של התנגשות חוקים. חוקי הברזל (מהחזק לחלש):

  • Inline Style: עיצוב הכתוב בתוך התגית עצמה (למשל style="color: red"). חזק מאוד.
  • ID (#): מזהה ייחודי. חזק יותר מ-Class.
  • Class (.): מחלקה. חזקה יותר מתגית.
  • Tag (div, p): בחירת האלמנט עצמו. הדרך החלשה ביותר.
1. מה נכון בנוגע ל-HTML ?
ניתוח התשובה: התשובה הנכונה היא ב'. חשוב לזכור למבחן ש-HTML לעולם אינה שפת תכנות, משום שאין לה יכולות לוגיות (חישובים, תנאי if, לולאות). תפקידה הבלעדי הוא "לסמן" (Markup) ולהגדיר לדפדפן את המבנה ההיררכי של המסמך (כותרות, פסקאות, קישורים).
2. קטע הקוד <...' > כתוב במסמך HTML. מה נכון בנוגע לו ולאלמנט xxx?
ניתוח התשובה: כאשר אנו רואים בתוך סוגריים משולשים מבנה של name="value" (למשל class="container" או src="image.jpg"), המילה השמאלית (xxx) היא המאפיין (Attribute). מאפיינים תמיד נכתבים בתוך תגית הפתיחה ותפקידם להרחיב את היכולות או המידע של אותה תגית.
3. מה נכון בנוגע לאלמנט DIV?
ניתוח התשובה: ה-DIV הוא "סוס העבודה" של מבנה האתר. הוא אלמנט Block (כלומר דוחף את שאר האלמנטים לשורה חדשה ומתפרס על כל הרוחב). אין לו משמעות סמנטית (קוראי מסך לא לומדים ממנו כלום), ולכן השימוש העיקרי בו הוא לעטוף אזורים בעמוד כדי שנוכל לעצב אותם יחד ב-CSS כגריד או פלקס.
8. מה נכון בנוגע ל- HTML5?
ניתוח התשובה: המהות של HTML5 במבחנים היא סמנטיקה ומולטימדיה טבעית. במקום להשתמש בעשרות divים, אנחנו משתמשים בתגיות בעלות משמעות. בנוסף, API חדשים כמו LocalStorage ותמיכה בטפסי HTML מתקדמים מאפיינים גרסה זו.
9. מה נכון בנוגע לעיצוב רספונסיבי (responsive design)?
ניתוח התשובה: הרעיון מאחורי Responsive Design הוא לכתוב קוד HTML אחד ויחיד. מי שעושה את עבודת ההתאמה הוא ה-CSS, וספציפית ה-Media Queries שמזהים את רוחב החלון (Viewport) ומסדרים את הגריד, הפונטים והגדלים מחדש (לרוב בעזרת Flexbox או Grid).
13. בהגדרות CSS3 יש קטע ... ואין הגדרות נוספות של צבע. איזה טקסט ... בקוד של HTML להלן יהיה בצבע אדום?
ניתוח התשובה: השאלה בוחנת את הבנת מנגנון ה-Specificity ב-CSS (ספציפיות/עדיפות). אם יש לך כמה הגדרות CSS שמנסות לשנות צבע של אותו טקסט (למשל, חוק אחד עבור פסקאות וחוק אחר עבור מחלקה), הדפדפן מחשב משקלים. אלמנט עם ID (מזהה ייחודי) תמיד ינצח אלמנט עם Class. ואלמנט עם סגנון בתוך ה-HTML (Inline style) ינצח את כולם.

2. לוגיקה ואינטראקציה בצד הלקוח (JavaScript & DOM)

שפת JavaScript ומאפייניה

JavaScript היא השפה שמעניקה חיים לדפדפן (צד הלקוח). היא שפה מפורשת (Interpreted) - קוד המקור נשלח ללקוח ומנוע ה-JS של הדפדפן מריץ אותו. מאפיין קריטי להבנה הוא ש-JS רצה על Single Thread (חוט ביצוע יחיד) בדפדפן, כלומר היא יכולה לבצע רק פעולה אחת בכל רגע נתון. לכן, כדי לא לתקוע את הממשק בזמן המתנה (למשל לתשובה משרת), היא משתמשת בתכנות אסינכרוני (Callbacks, Promises, Async/Await).

משתנים ופונקציות

הכרזת משתנים עברה מהפכה ב-ES6: בעבר השתמשנו ב-var אשר לו טווח חלות של פונקציה (Function scope) ונוטה לבעיות (Hoisting). כיום נהוג להשתמש ב-let למשתנים שערכם עשוי להשתנות בעתיד, וב-const לקבועים שלא ניתן להציב להם ערך מחדש. לשניהם יש טווח חלות של בלוק (Block scope - למשל בתוך לולאת for בלבד).

פונקציות ב-JS מוגדרות כ-"First-Class Citizens": משמעות הדבר היא שניתן להתייחס לפונקציה בדיוק כמו לכל משתנה אחר. אפשר לאחסן אותה במשתנה, להחזיר אותה מתוך פונקציה אחרת, והכי חשוב - להעביר אותה כארגומנט (פרמטר) לפונקציה אחרת (מה שמוכר כ-Callback Function).

ה-DOM ו-Fetch API

ה-DOM (Document Object Model) הוא בעצם ממשק תכנות (API) שהדפדפן מייצר. הוא קורא את קובץ ה-HTML הסטטי והופך אותו לעץ היררכי של אובייקטים (Nodes) השמור בזיכרון המחשב. JS משתמשת בעץ הזה כדי לתקשר עם התצוגה: לשנות טקסטים, להוסיף אלמנטים, לקרוא ערכים של טפסים ולשנות CSS בזמן אמת, הכל ללא רענון העמוד.

מתודת fetch היא הסטנדרט המודרני לביצוע בקשות AJAX (קריאות שרת ברקע). היא עובדת באופן אסינכרוני (כדי לא לעצור את חוט הביצוע היחיד) ומחזירה Promise - אובייקט שמייצג הבטחה להשלמה עתידית של פעולת הרשת (בהצלחה או בכישלון).

4. מה נכון בנוגע לשפת תכנות JS) Javascript)
ניתוח התשובה: JS היא לב האינטראקטיביות ברשת. אין לה קשר לשפת Java (למרות השם). היא רצה בדפדפן, קוראת את ה-DOM, מאזינה לאירועים של המשתמש (קליקים, הקלדות) ומבצעת תקשורת רשת (Fetch) ללא צורך בקומפילציה, אלא באמצעות Interpreter המובנה בדפדפן.
5. מה נכון בנוגע למשתנים ב-Javascript
ניתוח התשובה: JS היא שפה Weakly Typed (טיפוסיות חלשה), לכן לא רושמים int או string, אלא רק let/const. הסיבה המרכזית ש-let ו-const החליפו את var היא ניהול זיכרון נכון יותר והימנעות מבאגים עקב כך שהם חיים רק בתוך הבלוק (`{ }`) שבו הוגדרו.
6. מה נכון בנוגע לפונקציות של Javascript
ניתוח התשובה: המושג "First-Class Citizen" הוא קריטי ב-JS. מכיוון שפונקציה היא סוג של אובייקט, אפשר לעשות איתה הכל. יכולת זו היא מה שמאפשרת לנו לכתוב קוד אסינכרוני - אנחנו מעבירים פונקציה כארגומנט (Callback), והיא מופעלת רק מתי שהפעולה הסתיימה (למשל סיום טיימר או קבלת מידע מהשרת).
14. מה נכון בנוגע ל-DOM?
ניתוח התשובה: ה-DOM הוא "גשר". מצד אחד יש לנו קובץ טקסט סטטי (HTML). מצד שני יש לנו שפת תכנות חיה (JS). ה-DOM הוא המבנה התלת-ממדי (עץ יוחסין של הורה, ילדים, ואחים) שהדפדפן בונה כדי ש-JS תוכל לאתר אלמנטים (למשל `document.getElementById`) ולבצע בהם שינויים באופן דינמי.
15. מהו התפקיד של מתודת JavaScript-fetch
ניתוח התשובה: ה-Fetch API הוא המנגנון העדכני (שהחליף את XMLHttpRequest) כדי שצד הלקוח יוכל לתקשר עם ה-Backend. הוא אסינכרוני כדי שהמשתמש יוכל להמשיך להשתמש באתר בזמן שהנתונים (לרוב JSON) נמשכים ברקע מהשרת.

3. ניהול מצב, נתונים ותקשורת (Cookies, Session & JSON)

בעיית ה-Stateless ב-HTTP וניהול מצב (State)

פרוטוקול התקשורת HTTP הוא Stateless (חסר מצב). פירוש הדבר הוא שהשרת מעבד בקשה מהלקוח, מחזיר תשובה, ו"שוכח" ממנו לחלוטין. הוא לא יודע אם בקשה שניה הגיעה מאותו לקוח או מלקוח חדש. כדי לאפשר התחברות לחשבון אישי או עגלת קניות, אנחנו זקוקים למנגנוני ניהול מצב: Cookies ו-Session.

Cookies לעומת Session

Cookies (עוגיות): קבצי טקסט קטנים (עד כ-4KB) שנשמרים אצל הלקוח (בדפדפן). העוגיות נשלחות באופן אוטומטי ב-Header של כל בקשת HTTP לשרת המשויך אליהן. משום שהן נשמרות אצל הלקוח (והוא יכול לקרוא או לערוך אותן בקלות), הכלל החשוב ביותר הוא שלעולם לא שומרים בהן מידע רגיש (כמו סיסמאות או פרטי כרטיס אשראי). השימוש הקלאסי הוא העדפות משתמש (שפה, תצוגה) או שמירת Session ID.

Session (הפעלה): מנגנון בו הנתונים הרגישים נשמרים בצד השרת (בזיכרון, בטבלה או בבסיס נתונים מהיר כמו Redis). תהליך העבודה: הלקוח מתחבר -> השרת מקצה לו רשומה ומידע רגיש אצלו, ויוצר "תג מזהה" אקראי (Session ID) -> השרת שולח ללקוח רק את ה-ID לשמירה כעוגייה. בבקשה הבאה הלקוח שולח רק את התג, והשרת מזהה אותו בבטחה. המידע אינו חשוף ללקוח, וה-Session בדרך כלל נמחק בעת סגירת הדפדפן (אלא אם הוגדר אחרת).

פורמט JSON

JSON (JavaScript Object Notation): הפורמט הסטנדרטי להעברת נתונים ברשת מודרנית (החליף את ה-XML המסורבל). זהו טקסט טהור הבנוי על ארכיטקטורת מפתח-ערך (Key-Value). למרות שמו, JSON עצמאי מ-JS וכל שפות צד-השרת (Python, C#, PHP) יודעות לעבוד איתו.

כלל תחביר קריטי: ב-JSON, כל המפתחות (Keys) חייבים להיות עטופים במרכאות כפולות בלבד (למשל: {"id": 1, "name": "Tal"}). ערכים מספריים או בוליאניים אינם זקוקים למרכאות.

7. מה נכון בנוגע ל- JSON?
ניתוח התשובה: המטרה העיקרית של JSON היא לייצג אובייקטים מורכבים במחרוזת טקסט רגילה כדי שניתן יהיה לשלוח אותם דרך רשת האינטרנט בפרוטוקול HTTP, ושני הצדדים (שרת ולקוח) יוכלו לפרש אותם בקלות.
10. מה נכון בנוגע ל-cookies?
ניתוח התשובה: ההגדרה המדויקת של Cookies - קבצים קטנים בדפדפן הלקוח. החסרון הגדול שלהן הוא אבטחה, מכיוון שהלקוח יכול לראותן דרך כלי המפתחים (F12) ולשנותן, וכן משום שהן מצורפות אוטומטית לכל בקשה לאותו דומיין, מה שמייצר תקורה על הרשת.
12. מה נכון בנוגע ל-JSON?
ניתוח התשובה: שאלה זו מתמקדת בדקויות התחביר (Syntax). בעוד שבאובייקט רגיל ב-JS מותר לכתוב מפתח ללא מירכאות (כמו `{ age: 24 }`), ב-JSON תקין ורשמי חובה להשתמש במירכאות כפולות בלבד (`{ "age": 24 }`). בנוסף, JSON אינו תומך בהעברת פונקציות, אלא רק נתונים סטטיים.
17. מה נכון בנוגע ל-session ?
ניתוח התשובה: ההבדל בין עוגייה לסשן הוא מיקום שמירת הנתונים האמיתיים. ב-Session, הנתונים המלאים (מי מחובר, מה יש בעגלה, הרשאות מנהל) מאובטחים לחלוטין בשרת. הלקוח מחזיק רק מפתח קטן חסר משמעות בעצמו שנקרא Session ID המשמש ככרטיס כניסה לקריאת הנתונים בשרת.

4. ארכיטקטורת רשת, Backend ו-Node.js

מהי אפליקציית Web?

אפליקציית Web היא תוכנה המבוססת על מודל לקוח-שרת (Client-Server). צד הלקוח (Frontend) רץ בדפדפן ומשמש כממשק המשתמש (UI). הלקוח יוזם בקשות (Requests) באמצעות פרוטוקול התקשורת HTTP/HTTPS אל צד השרת (Backend). השרת, שהוא מחשב מרוחק הזמין ברשת, מעבד את הבקשה ומחזיר תגובה (Response) המכילה HTML/CSS או נתוני JSON טהורים המוצגים חזרה בדפדפן. לא נדרשת התקנת תוכנה על המחשב (בניגוד ל-Desktop Application).

פיתוח צד שרת (Backend)

ה-Backend הוא ה"מוח" המוסתר של האפליקציה. בעוד שהמשתמש הקצה יכול לקרוא ולראות את כל קוד ה-Frontend (על ידי לחיצה על "הצג מקור" בדפדפן), קוד ה-Backend מאובטח ורץ בשרת. תפקידיו המרכזיים כוללים: לוגיקה עסקית (חישובים, קבלת החלטות), אימות והרשאות משתמשים (Authentication), אבטחת מידע, והחשוב מכל - תקשורת מול בסיסי נתונים (Databases).

המהפכה של Node.js וניהול אסינכרוני

עד להופעת Node.js, JavaScript רצה אך ורק בדפדפן. Node.js היא סביבת ריצה (Runtime Environment) הכתובה ב-C++ ולוקחת את מנוע ה-V8 החזק של חברת גוגל, ומאפשרת להריץ בעזרתו קוד JS על השרת המרוחק. (חשוב למבחן: Node.js היא לא שפת תכנות חדשה!).

ארכיטקטורת Node.js: שרתי עבר (כמו Apache) נהגו לפתוח חוט ריצה חדש (Thread) בזיכרון לכל משתמש שמתחבר, מה שהכביד על המערכת. Node.js עובד עם Single Thread (חוט ביצוע יחיד) ומבוסס אירועים (Event-driven).

איך שרת עם חוט יחיד מטפל באלפי משתמשים מבלי לקרוס? התשובה היא Non-Blocking I/O (תקשורת אסינכרונית). כל פעולה "כבדה" שלוקחת זמן (כמו שאילתה למסד נתונים או קריאת קובץ מהדיסק) נשלחת לביצוע ברקע. בינתיים, השרת פנוי לקבל משתמשים חדשים. כאשר מסד הנתונים מסיים לשלוף את הנתונים, הוא שולח אירוע (Callback) ל-Node שממשיך מאותה נקודה. מכאן עולה שכל פעולה מול מסד נתונים חייבת להיות אסינכרונית.

11. מה נכון בנוגע ליישום Web?
ניתוח התשובה: מהות אפליקציית ה-Web היא ההפרדה בין צד הלקוח שמציג ממשק למשתמש (באמצעות דפדפן וטכנולוגיות WEB), לבין צד השרת שמעבד ושומר נתונים. התקשורת ביניהם מתבצעת על גבי רשת האינטרנט בפרוטוקול HTTP.
16. מה נכון בנוגע Node.js
ניתוח התשובה: המושגים שחובה לחפש בתשובות על Node.js הם: סביבת ריצה (לא שפת תכנות), צד-שרת, Single Thread, אסינכרוני מבוסס אירועים.
18. מה נכון בנוגע לשימוש בבסיסי נתונים ב-Node.js
ניתוח התשובה: אם היינו ניגשים למסד הנתונים בצורה סינכרונית ב-Node.js (שהוא כאמור בעל חוט ביצוע יחיד), כל האתר היה קופא ואף משתמש אחר לא היה מקבל מענה עד לשליפת הנתונים. לכן, הגישה היא תמיד אסינכרונית.
19. מה נכון בנוגע לפיתוח backend?
ניתוח התשובה: בעוד ש-Frontend דואג לממשק, ה-Backend הוא מרכז העצבים של האפליקציה שפועל בבטחה מאחורי הקלעים בשרת ומנהל את המידע האמיתי של המערכת.
20. מה נכון בנוגע לאפליקציית WEB ?
ניתוח התשובה: ההגדרה של אפליקציית WEB (שאלה שחוזרת על עצמה במבחן בניסוחים קלים) נועדה לוודא שאתה מבין שאין התקנה מקומית למשתמש (מלבד הדפדפן עצמו), והחיבור נעשה דרך מודל לקוח ושרת מרוחק.
